## 4. 基础知识铺垫

### 4.1 什么是公共表表达式（CTE）

CTE（Common Table Expression）是 SQL 的一种「临时命名查询」。基本语法：

```sql
WITH 临时表名 AS (
  SELECT ...
)
SELECT * FROM 临时表名;
```

CTE 的好处是让复杂查询更易读：把一个大查询拆成几个命名步骤，而不是套一层又一层的子查询。

### 4.2 什么是递归 CTE（`WITH RECURSIVE`）

递归 CTE 是 CTE 的特殊形式，允许查询**引用自身**，实现循环迭代的效果。就像编程里的递归函数，能处理树形或图形结构的数据。

基本结构：

```sql
WITH RECURSIVE 结果集名 AS (
  -- ①基础情形（Anchor）：不引用自身，给出初始值
  SELECT 初始行
  
  UNION ALL
  
  -- ②递归步骤（Recursive Step）：引用上一轮的结果
  SELECT 新行
  FROM 数据表
  JOIN 结果集名 ON 连接条件    ← 这里引用了自身！
)
SELECT * FROM 结果集名;
```

**执行过程（以祖先查询为例）**：

第 0 轮（基础情形）：找到成员「张三」（depth=0）

```
结果集 = [{ id: "A", name: "张三", depth: 0, parentId: null }]
```

第 1 轮（递归步骤）：找到「张三」的父母（depth=1）

```
JOIN parent_child pc ON pc.parent_id = m.id   ← 找父母
JOIN ancestors a ON pc.child_id = a.id        ← a 就是上一轮的结果（张三）
→ 结果集 += [{ id: "B", name: "张大山", depth: 1, parentId: "A" }]
```

第 2 轮：找到「张大山」的父母（depth=2）

```
a = [张大山（depth=1）]
→ 结果集 += [{ id: "C", name: "张老爷", depth: 2, parentId: "B" }]
```

…直到没有父母可找（上一轮没有新结果），递归终止。

最终合并所有轮次的结果：

```
[张三（0）, 张大山（1）, 张老爷（2）, ...]
```

> **关键直觉**：`WITH RECURSIVE` 就像数据库里的「一直往上找父母，直到找不到为止」。普通 SQL 只能做固定深度的查询（如 JOIN 三次查三代），递归 CTE 能处理任意深度的树。

### 4.3 什么是「图」和「邻接表」

「图」（Graph）是由「节点」（Vertex）和「边」（Edge）组成的数据结构。家族关系是一张图：
- 节点：每个家族成员
- 边：两人之间的亲子或婚姻关系（无方向，因为亲缘是双向的）

「邻接表」（Adjacency List）是图的一种表示方法：用一个 Map，每个节点对应其邻居节点的集合。

```
节点A（张三）→ 邻居：[张大山（父）, 李四（子）, 王五（配偶）]
节点B（张大山）→ 邻居：[张三（子）, 张老爷（父）]
...
```

本项目用 `Map<string, Set<string>>` 实现邻接表：

```typescript
const adj = new Map<string, Set<string>>();
// adj.get("张三的ID") = Set { "张大山的ID", "李四的ID", "王五的ID" }
```

### 4.4 什么是 BFS（广度优先搜索）

BFS（Breadth-First Search，广度优先搜索）是图遍历的一种算法，从起始节点出发，**先访问所有距离为1的邻居，再访问距离为2的邻居，以此类推**。

用「路径查找」来理解：

```
起点：张三
目标：某位 5 代祖先

第1步探索（张三的直接邻居）：
  队列 = [张三的父, 张三的母, 张三的配偶]

第2步探索（上述三人的邻居）：
  队列 = [张三父的父, 张三父的母, 张三母的父, ...]

第3步探索...
...

直到某个被探索的节点 = 目标节点，返回路径
```

BFS 找到的路径一定是**最短路径**（步数最少）。

BFS 需要一个「队列」（先进先出）数据结构，用 JavaScript 的数组实现：

```typescript
const queue = [起始节点];
while (queue.length > 0) {
  const current = queue.shift();  // 从队列头取出
  // 把 current 的邻居加入队列尾部
  for (const neighbor of getNeighbors(current)) {
    queue.push(neighbor);
  }
}
```

### 4.5 什么是 ECharts

ECharts 是百度开源、现归属 Apache 的 JavaScript 图表库，支持折线图、柱状图、地图、树图等几十种图表类型。本项目用它渲染「树形图」（Tree Chart）。

ECharts 的工作方式：
1. 创建 ECharts 实例（绑定到一个 DOM div）
2. 调用 `setOption` 传入配置对象（数据 + 样式）
3. ECharts 用 Canvas 或 SVG 渲染图表

在 React 中使用 ECharts 需要：
- 用 `useRef` 持有 DOM 节点
- 在 `useEffect` 里初始化 ECharts 实例（因为要操作真实 DOM）
- 返回 cleanup 函数处理销毁

### 4.6 什么是「动态导入」（`dynamic`）

ECharts 是一个很大的库（约 2MB）。如果在服务端渲染时加载它，会出现错误（ECharts 依赖浏览器 API，如 `window`、`document`，服务端没有这些）。

Next.js 的 `dynamic` 函数实现「动态导入」：

```typescript
const EChartsTree = dynamic(
  () => import("@/components/tree/echarts-tree").then((m) => m.EChartsTree),
  { ssr: false }  // 禁用服务端渲染
);
```

`ssr: false` 告诉 Next.js：这个组件只在浏览器里加载，不在服务器上运行。

### 5.1 步骤 1：`lib/queries/ancestors.ts` — 递归祖先查询 【开发者甲】

#### A. 动机

Prisma 的 `findMany` 没有办法表达「向上递归查所有层级的祖先」这种查询。这时需要用 `prisma.$queryRaw` 执行原生 SQL，写 `WITH RECURSIVE` 语句。

#### B. 完整代码

*文件 `lib/queries/ancestors.ts`：*

```typescript
import { prisma } from "../db";

export interface AncestorNode {
  id: string;
  name: string;
  gender: string;
  birthYear: number | null;
  deathYear: number | null;
  generation: number | null;
  depth: number;       // 距离起始成员的层数（0=本人，1=父母，2=祖父母，…）
  parentId: string | null;  // 在查询结果中指向上一层的节点ID（非数据库亲子关系）
}

/**
 * 递归祖先查询（Recursive CTE）
 * 输入成员 ID，向上追溯所有历代祖先
 */
export async function getAncestors(memberId: string): Promise<AncestorNode[]> {
  const result = await prisma.$queryRaw<AncestorNode[]>`
    WITH RECURSIVE ancestors AS (
      -- 基础情形：起始成员（depth=0）
      SELECT
        m.id,
        m.name,
        m.gender,
        m.birth_year   AS "birthYear",
        m.death_year   AS "deathYear",
        m.generation,
        0              AS depth,
        NULL::text     AS "parentId"
      FROM members m
      WHERE m.id = ${memberId}

      UNION ALL

      -- 递归步骤：向上找父母
      SELECT
        m.id,
        m.name,
        m.gender,
        m.birth_year   AS "birthYear",
        m.death_year   AS "deathYear",
        m.generation,
        a.depth + 1    AS depth,
        a.id           AS "parentId"
      FROM members m
      JOIN parent_child pc ON pc.parent_id = m.id
      JOIN ancestors a     ON pc.child_id  = a.id
    )
    SELECT * FROM ancestors ORDER BY depth, name
  `;
  return result;
}

/**
 * 递归后代查询（Recursive CTE）
 * 输入成员 ID，向下追溯所有历代后代
 */
export async function getDescendants(memberId: string): Promise<AncestorNode[]> {
  const result = await prisma.$queryRaw<AncestorNode[]>`
    WITH RECURSIVE descendants AS (
      SELECT
        m.id, m.name, m.gender,
        m.birth_year AS "birthYear", m.death_year AS "deathYear",
        m.generation,
        0            AS depth,
        NULL::text   AS "parentId"
      FROM members m
      WHERE m.id = ${memberId}

      UNION ALL

      SELECT
        m.id, m.name, m.gender,
        m.birth_year AS "birthYear", m.death_year AS "deathYear",
        m.generation,
        d.depth + 1  AS depth,
        d.id         AS "parentId"
      FROM members m
      JOIN parent_child pc ON pc.child_id  = m.id
      JOIN descendants d   ON pc.parent_id = d.id
    )
    SELECT * FROM descendants ORDER BY depth, name
  `;
  return result;
}
```

#### C. 祖先查询 SQL 的逐行解析

**基础情形**（查询起点）：

```sql
SELECT
  m.id, m.name, m.gender,
  m.birth_year AS "birthYear",  -- 列名映射（数据库下划线 → TypeScript 驼峰）
  m.death_year AS "deathYear",
  m.generation,
  0         AS depth,           -- 起始成员深度为 0
  NULL::text AS "parentId"      -- 起始成员没有「在结果集中的父节点」
FROM members m
WHERE m.id = ${memberId}        -- ${...} 是 Prisma 的参数化语法，防止 SQL 注入
```

**递归步骤**（查父母）：

```sql
SELECT
  m.id, m.name, m.gender,
  m.birth_year AS "birthYear",
  m.death_year AS "deathYear",
  m.generation,
  a.depth + 1  AS depth,        -- 父母的深度 = 子女的深度 + 1
  a.id         AS "parentId"    -- 在结果集里，父母的「结果父节点」就是该子女
FROM members m
JOIN parent_child pc ON pc.parent_id = m.id   -- 找父母：parent_id 指向父母的 ID
JOIN ancestors a     ON pc.child_id  = a.id   -- 连接上一轮结果：child_id 指向上一轮的成员
```

理解 JOIN 的方向：
- `parent_child` 表：`parent_id` 是父母，`child_id` 是子女
- `pc.parent_id = m.id`：找出所有把 `m`（某成员）作为父母的亲子记录，即 m 是父母
- `pc.child_id = a.id`：这个亲子记录里的子女是上一轮结果里的某个成员

所以这条 JOIN 的意思是：「找出上一轮结果中每个人的父母」。

**后代查询的区别**：方向相反！

```sql
-- 祖先查询（向上）：
JOIN parent_child pc ON pc.parent_id = m.id   -- m 是父母（要找的是父母这一侧）
JOIN ancestors a     ON pc.child_id  = a.id   -- a 是子女（从子女找父母）

-- 后代查询（向下）：
JOIN parent_child pc ON pc.child_id  = m.id   -- m 是子女（要找的是子女这一侧）
JOIN descendants d   ON pc.parent_id = d.id   -- d 是父母（从父母找子女）
```

> **反例对比**：如果不用递归 CTE，而是用 Prisma 的 `include` 嵌套，只能写固定深度，如最多 5 层。实际家谱可能有 30+ 代，必须用递归 CTE 处理任意深度。

### 5.2 步骤 2：`lib/queries/relationship.ts` — BFS 亲缘路径 【开发者甲】

#### A. 完整代码

*文件 `lib/queries/relationship.ts`：*

```typescript
import { prisma } from "../db";

export interface RelationEdge {
  memberA: string;  // 成员 A 的 ID
  memberB: string;  // 成员 B 的 ID（两者之间有直接亲缘关系）
}

/**
 * 获取族谱内全部亲缘邻接关系（亲子 + 婚姻，双向）
 */
export async function getAllRelationEdges(genealogyId: string): Promise<RelationEdge[]> {
  const result = await prisma.$queryRaw<RelationEdge[]>`
    -- 亲子关系（父 → 子）
    SELECT pc.parent_id AS "memberA", pc.child_id AS "memberB"
    FROM parent_child pc
    JOIN members m ON m.id = pc.parent_id
    WHERE m.genealogy_id = ${genealogyId}

    UNION

    -- 亲子关系（子 → 父，反向）
    SELECT pc.child_id AS "memberA", pc.parent_id AS "memberB"
    FROM parent_child pc
    JOIN members m ON m.id = pc.child_id
    WHERE m.genealogy_id = ${genealogyId}

    UNION

    -- 婚姻关系（m1 → m2）
    SELECT mg.member1_id AS "memberA", mg.member2_id AS "memberB"
    FROM marriages mg
    JOIN members m ON m.id = mg.member1_id
    WHERE m.genealogy_id = ${genealogyId}

    UNION

    -- 婚姻关系（m2 → m1，反向）
    SELECT mg.member2_id AS "memberA", mg.member1_id AS "memberB"
    FROM marriages mg
    JOIN members m ON m.id = mg.member2_id
    WHERE m.genealogy_id = ${genealogyId}
  `;
  return result;
}

/**
 * BFS 路径查找：在应用层找两人之间最短亲缘路径
 */
export function findRelationPath(
  edges: RelationEdge[],
  startId: string,
  endId: string
): string[] | null {
  if (startId === endId) return [startId];

  // 第1步：构建邻接表
  const adj = new Map<string, Set<string>>();
  for (const { memberA, memberB } of edges) {
    if (!adj.has(memberA)) adj.set(memberA, new Set());
    adj.get(memberA)!.add(memberB);
  }

  // 第2步：BFS
  const visited = new Set<string>([startId]);
  const queue: { id: string; path: string[] }[] = [
    { id: startId, path: [startId] }
  ];

  while (queue.length > 0) {
    const { id, path } = queue.shift()!;         // 从队列头取出当前节点
    const neighbors = adj.get(id) ?? new Set();

    for (const neighbor of neighbors) {
      if (visited.has(neighbor)) continue;        // 已访问过，跳过（避免循环）
      const newPath = [...path, neighbor];
      if (neighbor === endId) return newPath;     // 找到终点，返回路径
      visited.add(neighbor);
      queue.push({ id: neighbor, path: newPath }); // 加入队列继续探索
    }
  }

  return null;  // 没有路径
}
```

#### B. SQL 的解析：UNION 和双向边

`getAllRelationEdges` 用 4 个 `UNION` 合并出所有边：
- 亲子关系正向（父→子）
- 亲子关系反向（子→父）
- 婚姻关系正向（m1→m2）
- 婚姻关系反向（m2→m1）

为什么要双向？BFS 在图里搜索时，需要从任意节点找到其邻居，所以每条关系必须表示为双向边。

#### C. BFS 代码的解析

**邻接表构建**：

```typescript
const adj = new Map<string, Set<string>>();
for (const { memberA, memberB } of edges) {
  if (!adj.has(memberA)) adj.set(memberA, new Set());
  adj.get(memberA)!.add(memberB);
}
```

遍历所有边，把「从 A 可以到 B」存入邻接表。

**BFS 主循环**：

```typescript
const queue = [{ id: startId, path: [startId] }];
while (queue.length > 0) {
  const { id, path } = queue.shift();  // 队列头部出队
  for (const neighbor of adj.get(id)) {
    if (visited.has(neighbor)) continue; // 跳过已访问
    const newPath = [...path, neighbor]; // 路径 = 当前路径 + 新邻居
    if (neighbor === endId) return newPath; // 找到了！
    visited.add(neighbor);
    queue.push({ id: neighbor, path: newPath }); // 入队
  }
}
```

每个队列节点不仅存当前成员 ID，还存「从起点到这里走过的路径」。找到终点时直接返回路径，不需要回溯。

> **反例对比**：如果用 DFS（深度优先搜索，用栈而不是队列），可能会找到一条很长的路径，而不是最短的。BFS 保证第一次找到终点时走的路径就是最短的（步数最少）。

### 5.3 步骤 3：`lib/tree-builder.ts` — 扁平节点转树结构 【开发者乙】

#### A. 动机

递归 CTE 返回的是一个「扁平」的节点列表（每条记录包含 `parentId` 字段）。ECharts 需要「嵌套」的树结构（`{ name, children: [{...}, {...}] }`）。`buildTree` 做这个转换。

#### B. 完整代码

*文件 `lib/tree-builder.ts`：*

```typescript
interface RawNode {
  id: string;
  name: string;
  gender: string;
  birthYear: number | null;
  deathYear: number | null;
  generation: number | null;
  depth: number;
  parentId: string | null;
}

interface TreeNode {
  name: string;
  value: string;         // 鼠标悬停时显示的信息
  children: TreeNode[];
  itemStyle?: { color: string };
}

export function buildTree(nodes: RawNode[], rootId: string): TreeNode | null {
  // 第1步：把每个节点转换为 TreeNode 并存入 Map
  const map = new Map<string, TreeNode & { _id: string }>();

  for (const n of nodes) {
    const lifespan = n.birthYear
      ? n.deathYear
        ? `${n.birthYear}–${n.deathYear}`  // 有生有卒
        : `${n.birthYear}–`                 // 有生无卒（还在世或不详）
      : "";
    const gen = n.generation ? `第${n.generation}代` : "";

    map.set(n.id, {
      _id: n.id,
      name: n.name,
      value: [n.gender === "M" ? "♂" : "♀", lifespan, gen].filter(Boolean).join(" · "),
      children: [],
      itemStyle: { color: n.gender === "M" ? "#3b82f6" : "#ec4899" },  // 蓝色男，粉色女
    });
  }

  // 第2步：根据 parentId 建立父子关系
  for (const n of nodes) {
    if (n.parentId && map.has(n.parentId) && map.has(n.id)) {
      map.get(n.parentId)!.children.push(map.get(n.id)!);
    }
  }

  return map.get(rootId) ?? null;  // 返回根节点（含完整树形结构）
}
```

#### C. 算法解析

**从扁平列表到嵌套树的过程**：

输入（CTE 返回的扁平列表）：
```
[
  { id: "A", name: "张三",   depth: 0, parentId: null },
  { id: "B", name: "张大山", depth: 1, parentId: "A" },
  { id: "C", name: "张老爷", depth: 2, parentId: "B" },
]
```

第1步之后（Map）：
```
{
  "A": { name: "张三",   children: [] },
  "B": { name: "张大山", children: [] },
  "C": { name: "张老爷", children: [] },
}
```

第2步之后（建立父子关系）：
```
{
  "A": { name: "张三",   children: [
    { name: "张大山", children: [
      { name: "张老爷", children: [] }
    ]}
  ]},
  ...
}
```

返回 `map.get("A")` 就是完整的嵌套树结构，ECharts 可以直接渲染。

### 5.4 步骤 4：`components/tree/echarts-tree.tsx` — ECharts 树形图 【开发者乙】

#### A. 完整代码

*文件 `components/tree/echarts-tree.tsx`：*

```typescript
"use client";
import { useEffect, useRef } from "react";
import * as echarts from "echarts";

interface TreeNode {
  name: string;
  value?: string;
  children?: TreeNode[];
  itemStyle?: object;
}

interface EChartsTreeProps {
  data: TreeNode;
  height?: number;
  orient?: "LR" | "TB" | "RL" | "BT";  // 树的方向
}

export function EChartsTree({ data, height = 500, orient = "TB" }: EChartsTreeProps) {
  const containerRef = useRef<HTMLDivElement>(null);
  const chartRef = useRef<echarts.ECharts | null>(null);

  useEffect(() => {
    if (!containerRef.current) return;

    // 第一次渲染时初始化 ECharts 实例
    if (!chartRef.current) {
      chartRef.current = echarts.init(containerRef.current);
    }

    const option: echarts.EChartsOption = {
      tooltip: {
        trigger: "item",
        formatter: (params: any) => {
          const d = params.data;
          return `<b>${d.name}</b>${d.value ? `<br/>${d.value}` : ""}`;
        },
      },
      series: [
        {
          type: "tree",
          data: [data],
          orient,                    // 方向：TB=从上到下，BT=从下到上，LR=从左到右
          symbol: "circle",
          symbolSize: 10,
          roam: true,                // 允许缩放和平移
          initialTreeDepth: 4,       // 初始展开 4 层
          label: {
            position: orient === "LR" || orient === "RL" ? "left" : "top",
            fontSize: 12,
            color: "#374151",
          },
          leaves: {
            label: {
              position: orient === "LR" || orient === "RL" ? "right" : "bottom",
            },
          },
          lineStyle: { color: "#d1d5db", width: 1.5 },
          itemStyle: { color: "#10b981", borderColor: "#059669", borderWidth: 2 },
          emphasis: {
            focus: "descendant",    // 悬停时高亮当前节点及所有后代
            itemStyle: { color: "#059669" },
          },
          expandAndCollapse: true,   // 点击节点可折叠/展开
          animationDuration: 300,
        },
      ],
    };

    chartRef.current.setOption(option);

    const handleResize = () => chartRef.current?.resize();
    window.addEventListener("resize", handleResize);
    return () => {
      window.removeEventListener("resize", handleResize);
    };
  }, [data, orient]);

  return <div ref={containerRef} style={{ width: "100%", height }} />;
}
```

#### B. 关键设计解析

**为什么用 `chartRef` 而不是每次重建实例**：

```typescript
if (!chartRef.current) {
  chartRef.current = echarts.init(containerRef.current);
}
chartRef.current.setOption(option);
```

ECharts 实例的初始化（`echarts.init`）很重，需要创建 Canvas 等资源。`data` 更新时，我们只调用 `setOption` 更新配置，复用已有实例。`chartRef` 是 `ref`（不触发重渲染），完美适合持有这种「外部库实例」。

**树的方向（`orient`）**：

本项目在祖先查询页用 `"BT"`（Bottom-Top，从下到上）：本人在最下方，祖先向上排列。在后代查询时用 `"TB"`（Top-Bottom）。

## 7. 本章小结

| 概念 | 一句话记忆 |
|------|-----------|
| `WITH RECURSIVE` | SQL 递归查询，处理树形/图形结构，可追溯任意深度 |
| 基础情形（Anchor） | 递归的起点，不引用自身 |
| 递归步骤 | 引用上一轮结果，向上/向下扩展一层 |
| BFS（广度优先搜索） | 用队列层层扩展，保证找到最短路径 |
| 邻接表 | `Map<节点ID, Set<邻居ID>>`，图的高效表示 |
| `buildTree` | 把带 `parentId` 的扁平列表转为 ECharts 所需的嵌套树 |
| `echarts.init` | 初始化 ECharts 实例（绑定到 DOM div） |
| `dynamic({ ssr: false })` | 只在浏览器加载组件，避免 ECharts 在服务端报错 |
