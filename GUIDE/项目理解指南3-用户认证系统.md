# 项目理解指南 3 — 用户认证系统

## 0. 前言

指南2建好了数据库结构。本章讲**用户认证系统**：如何注册账号、登录验证、保持登录状态、以及如何阻止未登录用户访问受保护页面。

认证系统涉及前后端配合：开发者甲写后端逻辑（密码加密、JWT生成），开发者乙写前端界面（登录表单、注册表单）。

读完本章，你应当能够：
1. 解释密码为什么不能明文存储，以及 bcrypt 哈希是怎么工作的
2. 解释 JWT（JSON Web Token）是什么，以及 Session 是如何维持的
3. 描述用户从点击「登录」到进入 Dashboard 的完整流程
4. 解释路由守卫（proxy.ts）如何阻止未登录用户

## 4. 基础知识铺垫

### 4.1 为什么密码不能明文存储

如果数据库里存的是明文密码 `"abc123"`，一旦数据库被盗，攻击者立刻知道所有用户的密码，甚至可以用来登录其他网站（很多人会复用密码）。

**哈希（Hash）** 是一种单向加密：把任意长度的文本变成固定长度的「摘要」，无法反推回原文。

```
bcrypt("abc123")  →  "$2b$10$xyz..." （每次结果不同）
bcrypt("abc123")  →  "$2b$10$abc..." （不同 hash，但都能验证）

// 验证：不是"解密"，而是重新计算后对比
bcrypt.compare("abc123", "$2b$10$xyz...")  →  true（匹配）
bcrypt.compare("abc456", "$2b$10$xyz...")  →  false（不匹配）
```

**bcrypt 的「加盐」**：每次哈希时生成随机的「盐」（Salt）混入密码，所以即使两个用户密码相同，哈希结果也不同。这防止了「彩虹表攻击」（预计算所有常见密码的哈希值）。

`bcrypt.hash(password, 10)` 中的 `10` 是「工作因子」（Cost Factor），越大越慢（抵抗暴力破解），10 是推荐的平衡值。

### 4.2 什么是 Session（会话）

HTTP 协议是无状态的：每次请求对服务器来说都是陌生的，服务器不知道「这个请求是刚才登录的那个用户发的」。

**Session（会话）** 机制解决这个问题：登录成功后，服务器给浏览器发一个「凭证」，浏览器每次请求都带上这个凭证，服务器就能识别用户。

### 4.3 什么是 JWT（JSON Web Token）

JWT（读作 jot）是一种轻量级的会话凭证格式。它由三部分组成，用点号分隔：

```
Header.Payload.Signature
```

示例（解码后）：
```json
// Header（算法信息）
{ "alg": "HS256", "typ": "JWT" }

// Payload（实际内容，如用户ID）
{ "sub": "clx1abc...", "name": "张三", "exp": 1234567890 }

// Signature（用 AUTH_SECRET 对前两部分签名，防篡改）
HMACSHA256(base64(header) + "." + base64(payload), "secret")
```

本项目的 JWT 存在浏览器的 **Cookie** 中，每次请求自动携带，服务器验证签名后就知道是哪个用户。

**JWT vs 传统 Session**：
- 传统 Session：服务器内存里存 Session 数据，Cookie 里只存 Session ID
- JWT：服务器不存任何东西，所有信息都在 Token 里（只要验证签名就能信任）

JWT 的好处：无状态，扩展容易；缺点：Token 被盗前无法主动失效（所以要设置短过期时间）。

### 4.4 什么是 NextAuth.js

NextAuth（现在叫 Auth.js）是 Next.js 生态中最流行的认证库。它处理了「认证」流程中最繁琐的部分：
- 创建登录/注册 API 端点
- 生成和验证 JWT
- 管理 Cookie
- 提供 `auth()` 函数（在服务端获取当前用户信息）
- 提供 `signIn()` / `signOut()` 函数（在客户端触发登录/注销）

本项目使用 v5（beta）版本，配置方式和 v4 有区别。

### 4.5 什么是「路由守卫」（Middleware）

路由守卫（也叫中间件，Middleware）是在请求到达页面**之前**先执行的代码。本项目用它做两件事：
1. 未登录用户访问任何受保护页面 → 重定向到 `/login`
2. 已登录用户访问 `/login`/`/register` → 重定向到 `/dashboard`

Next.js 15/16 中，中间件文件必须叫 `middleware.ts` 放在根目录，但本项目由于 Next.js 16 的 deprecation warning，改名为 `proxy.ts`（通过 `next.config` 配置）。

### 4.6 TypeScript 类型扩展是什么

NextAuth 的 `Session` 类型默认只有 `user.name`、`user.email`、`user.image`，没有 `user.id`。但我们的应用需要用 `session.user.id` 来查询数据库，所以需要告诉 TypeScript「Session 的 user 对象还有 id 字段」。

这通过 `types/next-auth.d.ts` 文件的「声明合并」（Declaration Merging）实现，是 TypeScript 的一种高级用法。

## 5. 逐步代码讲解

### 5.1 步骤 1：`lib/auth.ts` — NextAuth 配置 【开发者甲】

#### A. 动机

这是整个认证系统的核心配置。它告诉 NextAuth：
- 用户怎么登录（用户名+密码 vs OAuth）
- 验证通过后 JWT 里存什么信息
- Session 对象包含什么字段

#### B. 完整代码

*文件 `lib/auth.ts`：*

```typescript
import NextAuth from "next-auth";
import Credentials from "next-auth/providers/credentials";
import bcrypt from "bcryptjs";
import { prisma } from "./db";

export const { handlers, signIn, signOut, auth } = NextAuth({
  providers: [
    Credentials({
      name: "credentials",
      credentials: {
        username: { label: "用户名", type: "text" },
        password: { label: "密码", type: "password" },
      },
      async authorize(credentials) {
        if (!credentials?.username || !credentials?.password) return null;

        const user = await prisma.user.findUnique({
          where: { username: credentials.username as string },
        });
        if (!user) return null;

        const valid = await bcrypt.compare(
          credentials.password as string,
          user.password
        );
        if (!valid) return null;

        return { id: user.id, name: user.username, email: user.email };
      },
    }),
  ],
  callbacks: {
    async session({ session, token }) {
      if (token.sub) session.user.id = token.sub;
      return session;
    },
    async jwt({ token, user }) {
      if (user) token.sub = user.id;
      return token;
    },
  },
  pages: {
    signIn: "/login",
    error: "/login",
  },
  session: { strategy: "jwt" },
});
```

#### C. 这段代码做了什么

**`NextAuth({...})` 返回什么**

```typescript
export const { handlers, signIn, signOut, auth } = NextAuth({...});
```

`NextAuth` 函数接收配置，返回 4 个工具：
- `handlers`：HTTP 处理器（被 `app/api/auth/[...nextauth]/route.ts` 导出，处理 `/api/auth/*` 路径的请求）
- `signIn`：在服务端或客户端触发登录的函数
- `signOut`：触发注销的函数
- `auth`：在服务端获取当前 Session 的函数（相当于「我是谁」）

**`Credentials` 提供者**

```typescript
providers: [
  Credentials({
    credentials: {
      username: { label: "用户名", type: "text" },
      password: { label: "密码", type: "password" },
    },
    async authorize(credentials) { ... }
  })
]
```

`Credentials` 表示用「自定义用户名+密码」登录（相对于 OAuth 的 Google、GitHub 登录）。

`authorize` 函数是核心：NextAuth 收到登录请求后调用这个函数，传入用户填写的 `credentials`。
- 返回 `null`：表示验证失败，NextAuth 返回错误
- 返回一个对象：表示验证成功，这个对象就是「用户信息」，会被存到 JWT 里

**验证逻辑**：
1. 检查用户名和密码不为空
2. 用用户名在数据库里查找用户（`prisma.user.findUnique`）
3. 用 `bcrypt.compare` 对比输入的密码和数据库里的哈希值
4. 验证通过，返回 `{ id, name, email }`

**`callbacks`（回调函数）**

```typescript
callbacks: {
  async jwt({ token, user }) {
    if (user) token.sub = user.id;  // 把用户 ID 存入 JWT
    return token;
  },
  async session({ session, token }) {
    if (token.sub) session.user.id = token.sub;  // 把 JWT 里的 ID 放入 Session
    return session;
  },
}
```

- `jwt` 回调：每次生成/刷新 JWT 时调用。把用户 ID 存入 `token.sub`（sub 是 JWT 的标准字段，代表 subject/主体）
- `session` 回调：每次读取 Session 时调用。把 `token.sub` 放到 `session.user.id`，这样应用代码里用 `session.user.id` 就能拿到用户 ID

**`session: { strategy: "jwt" }`**

告诉 NextAuth 用 JWT 策略（无状态），不在服务器端存 Session 数据。

> **反例对比**：如果不设置 `jwt` 和 `session` callbacks，`session.user.id` 永远是 `undefined`，所有用 ID 查数据库的代码都会出错（因为 NextAuth 默认只在 session 里存 `name`、`email`、`image`，没有 `id`）。

### 5.2 步骤 2：`types/next-auth.d.ts` — TypeScript 类型扩展 【开发者甲】

#### A. 动机

虽然 `lib/auth.ts` 里的回调已经在运行时把 `user.id` 放到 Session，但 TypeScript 不知道这件事，它还以为 `session.user` 没有 `id` 字段，会在编译时报错。

#### B. 完整代码

*文件 `types/next-auth.d.ts`：*

```typescript
import "next-auth";

declare module "next-auth" {
  interface Session {
    user: {
      id: string;
      name?: string | null;
      email?: string | null;
      image?: string | null;
    };
  }
}
```

#### C. 这段代码做了什么

`declare module "next-auth"` 进入 `next-auth` 模块的「类型命名空间」。`interface Session` 利用 TypeScript 的「接口合并」（Declaration Merging）机制：同名接口的字段会被合并，所以这里定义的 `Session` 会和 `next-auth` 原有的 `Session` 类型合并，使得 `session.user.id` 有完整的类型提示，不再报错。

这一行把 `lib/auth.ts` 导出的所有内容重新从根目录导出，这样：

### 5.5 步骤 5：`app/api/users/register/route.ts` — 注册接口 【开发者甲】

#### A. 动机

注册账号不能用 NextAuth 处理（NextAuth 只处理登录流程），需要单独写一个 API 接口。这个接口接收用户名、邮箱和密码，验证后把密码哈希存入数据库。

#### B. 完整代码

*文件 `app/api/users/register/route.ts`：*

```typescript
import { NextRequest, NextResponse } from "next/server";
import bcrypt from "bcryptjs";
import { prisma } from "@/lib/db";

export async function POST(req: NextRequest) {
  const { username, email, password } = await req.json();

  if (!username || !email || !password) {
    return NextResponse.json({ error: "缺少必填字段" }, { status: 400 });
  }

  const exists = await prisma.user.findFirst({
    where: { OR: [{ username }, { email }] },
  });
  if (exists) {
    return NextResponse.json({ error: "用户名或邮箱已被注册" }, { status: 409 });
  }

  const hashed = await bcrypt.hash(password, 10);
  const user = await prisma.user.create({
    data: { username, email, password: hashed },
  });

  return NextResponse.json({ id: user.id, username: user.username });
}
```

#### C. 这段代码做了什么

**`await req.json()`**：解析请求体中的 JSON 数据，拿到前端发来的 `{ username, email, password }`。

**检查必填字段**：如果三个字段有任何一个缺失，返回 400 错误（Bad Request）。

**检查用户名/邮箱唯一性**：

```typescript
const exists = await prisma.user.findFirst({
  where: { OR: [{ username }, { email }] },
});
```

`OR` 条件：用户名**或**邮箱任意一个已存在都算冲突，返回 409 错误（Conflict）。

**哈希密码并创建用户**：

```typescript
const hashed = await bcrypt.hash(password, 10);
const user = await prisma.user.create({
  data: { username, email, password: hashed },
});
```

`bcrypt.hash(password, 10)`：工作因子 10，哈希计算需要约 100ms（故意慢，让暴力破解成本极高）。

**返回**：只返回 `id` 和 `username`，**绝不返回密码哈希**（即使是哈希值也不应该暴露给前端）。

> **反例对比**：如果直接 `prisma.user.create({ data: { username, email, password } })` 不哈希，数据库泄露时所有密码都是明文。bcrypt 哈希是最基本的安全实践，不可省略。

### 5.6 步骤 6：`proxy.ts` — 路由守卫 【开发者甲】

#### A. 动机

每个受保护的页面都需要验证登录状态，如果每个页面都自己检查，非常冗余。路由守卫在「所有请求到达页面之前」统一检查，是更优雅的解决方案。

#### B. 完整代码

*文件 `proxy.ts`：*

```typescript
import { auth } from "./auth";
import { NextResponse } from "next/server";

export default auth((req) => {
  const { nextUrl, auth: session } = req;
  const isLoggedIn = !!session;

  const isAuthPage =
    nextUrl.pathname.startsWith("/login") ||
    nextUrl.pathname.startsWith("/register");

  if (!isLoggedIn && !isAuthPage) {
    return NextResponse.redirect(new URL("/login", nextUrl));
  }
  if (isLoggedIn && isAuthPage) {
    return NextResponse.redirect(new URL("/dashboard", nextUrl));
  }
  return NextResponse.next();
});

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};
```

#### C. 这段代码做了什么

**`auth((req) => {...})` 包裹**

`auth` 函数来自 `lib/auth.ts`（NextAuth 导出的）。当它包裹一个函数时，会先解析 Cookie 里的 JWT，把解析出的 Session 放到 `req.auth`，然后调用被包裹的函数。

所以 `req.auth` 就是当前的登录状态：未登录时为 `null`，已登录时是 `{ user: { id, name, email } }`。

**逻辑判断**：

```typescript
const isLoggedIn = !!session;                          // session 非空则已登录
const isAuthPage = pathname.startsWith("/login") || ... // 是否是认证页

if (!isLoggedIn && !isAuthPage) → 重定向到 /login      // 未登录且不是认证页 → 去登录
if (isLoggedIn && isAuthPage)  → 重定向到 /dashboard   // 已登录但在认证页 → 去首页
return NextResponse.next()                              // 其他情况 → 正常放行
```

**`config.matcher`**：

```typescript
matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"]
```

这是一个正则表达式，意思是「除了 `api/`、`_next/static`（静态文件）、`_next/image`（图片）、`favicon.ico` 以外的所有路径都要经过这个守卫」。如果不排除 API 路由，认证 API（`/api/auth/...`）也会被拦截，造成死循环。

### 5.7 步骤 7：`app/(auth)/login/page.tsx` — 登录页面 【开发者乙】

#### A. 动机

这是用户看到的登录表单。它是客户端组件（有用户交互），调用 NextAuth 的 `signIn` 函数发送登录请求。

#### B. 完整代码

*文件 `app/(auth)/login/page.tsx`：*

```typescript
"use client";
import { useState } from "react";
import { signIn } from "next-auth/react";
import { useRouter } from "next/navigation";
import Link from "next/link";
import { Input } from "@/components/ui/input";
import { Button } from "@/components/ui/button";
import { BookOpen } from "lucide-react";

export default function LoginPage() {
  const router = useRouter();
  const [form, setForm] = useState({ username: "", password: "" });
  const [error, setError] = useState("");
  const [loading, setLoading] = useState(false);

  async function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
    setError("");
    setLoading(true);
    const res = await signIn("credentials", {
      username: form.username,
      password: form.password,
      redirect: false,
    });
    setLoading(false);
    if (res?.error) {
      setError("用户名或密码错误");
    } else {
      router.push("/dashboard");
      router.refresh();
    }
  }

  return (
    <div className="min-h-screen flex items-center justify-center bg-gradient-to-br from-emerald-50 to-zinc-100">
      <div className="w-full max-w-sm">
        {/* Logo 区域 */}
        <div className="flex flex-col items-center mb-8">
          <div className="bg-emerald-600 text-white p-3 rounded-2xl mb-3">
            <BookOpen size={28} />
          </div>
          <h1 className="text-2xl font-bold text-zinc-900">寻根溯源</h1>
          <p className="text-sm text-zinc-500 mt-1">族谱管理系统</p>
        </div>

        {/* 登录卡片 */}
        <div className="bg-white rounded-2xl shadow-sm border border-zinc-200 p-6">
          <h2 className="text-lg font-semibold text-zinc-800 mb-5">登录账号</h2>
          <form onSubmit={handleSubmit} className="space-y-4">
            <Input
              id="username"
              label="用户名"
              placeholder="请输入用户名"
              value={form.username}
              onChange={(e) => setForm({ ...form, username: e.target.value })}
              required
            />
            <Input
              id="password"
              label="密码"
              type="password"
              placeholder="请输入密码"
              value={form.password}
              onChange={(e) => setForm({ ...form, password: e.target.value })}
              required
            />
            {error && (
              <p className="text-sm text-red-500 bg-red-50 rounded-lg px-3 py-2">{error}</p>
            )}
            <Button type="submit" className="w-full" loading={loading}>
              登录
            </Button>
          </form>
          <p className="mt-4 text-center text-sm text-zinc-500">
            还没有账号？{" "}
            <Link href="/register" className="text-emerald-600 font-medium hover:underline">
              立即注册
            </Link>
          </p>
        </div>
      </div>
    </div>
  );
}
```

#### C. 这段代码做了什么

**`"use client"` 声明**

因为这个页面有 `useState`（状态管理）和表单提交等交互，必须是客户端组件。

**状态管理**：

```typescript
const [form, setForm] = useState({ username: "", password: "" });
const [error, setError] = useState("");
const [loading, setLoading] = useState(false);
```

三个状态：
- `form`：表单输入值
- `error`：错误提示信息
- `loading`：按钮的加载状态（提交时变为 true，防止重复提交）

**`handleSubmit` 函数**：

```typescript
const res = await signIn("credentials", {
  username: form.username,
  password: form.password,
  redirect: false,  // 不让 NextAuth 自动重定向，我们自己处理
});
```

`signIn` 来自 `next-auth/react`（客户端版本）。`redirect: false` 是关键：如果不设置，登录失败时 NextAuth 会跳转到错误页，设置后我们自己判断 `res.error` 来显示友好的错误提示。

**Tailwind CSS 样式**：

```html
<div className="min-h-screen flex items-center justify-center bg-gradient-to-br from-emerald-50 to-zinc-100">
```

这行 Tailwind 类名意思：
- `min-h-screen`：最小高度为视口高度（撑满全屏）
- `flex items-center justify-center`：使用 Flexbox 垂直+水平居中
- `bg-gradient-to-br`：向右下角的渐变背景
- `from-emerald-50 to-zinc-100`：从浅绿色到浅灰色的渐变

## 6. 端到端走查：用户点击「登录」按钮

| 步骤 | 文件/位置 | 发生了什么 |
|------|---------|-----------|
| ① | `login/page.tsx :: handleSubmit` | 用户点击登录按钮，`setLoading(true)` 显示 Loading 状态 |
| ② | `next-auth/react :: signIn()` | 发送 POST 到 `/api/auth/callback/credentials` |
| ③ | `app/api/auth/[...nextauth]/route.ts` | NextAuth handlers 收到请求 |
| ④ | `lib/auth.ts :: authorize()` | 查数据库找用户，bcrypt.compare 验证密码 |
| ⑤ | 验证成功路径 | authorize 返回 `{ id, name, email }` |
| ⑥ | `lib/auth.ts :: jwt callback` | `token.sub = user.id`，生成 JWT |
| ⑦ | NextAuth 内部 | JWT 序列化后存入 Cookie（`next-auth.session-token`） |
| ⑧ | `login/page.tsx` | `res.error` 为空，`router.push("/dashboard")` |
| ⑨ | `proxy.ts` | 请求 `/dashboard` 时，读 Cookie 里的 JWT，解析得 session，`isLoggedIn = true`，放行 |
| ⑩ | `(main)/dashboard/page.tsx` | 页面渲染，`auth()` 获取 session，显示用户数据 |

## 7. 本章小结

| 概念 | 一句话记忆 |
|------|-----------|
| bcrypt.hash | 密码单向哈希，无法反解，有随机盐 |
| bcrypt.compare | 验证密码，重新计算后对比，不解密 |
| JWT | 携带用户信息的签名字符串，存在 Cookie 里 |
| NextAuth `authorize` | 验证用户名密码的核心函数 |
| Session callbacks | 把用户 ID 从 JWT 传到 Session 对象 |
| `proxy.ts` | 路由守卫：未登录跳登录页，已登录跳首页 |
| `"use client"` | 声明客户端组件，可用 useState/事件处理 |
