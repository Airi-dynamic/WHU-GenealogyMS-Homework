## 4. 基础知识铺垫

### 4.1 什么是关系型数据库（PostgreSQL）

关系型数据库把数据存在**表（Table）**中，表像 Excel 表格：有列名（字段），有若干行（记录）。

PostgreSQL（简称 PG）是最流行的开源关系型数据库之一，支持复杂查询、事务、丰富的数据类型。本项目使用 18 版本。

```
表 members（成员表）：
┌──────────────────┬────────┬────────┬────────────┐
│ id               │ name   │ gender │ birth_year │
├──────────────────┼────────┼────────┼────────────┤
│ clx1abc...       │ 张三   │ M      │ 1920       │
│ clx2def...       │ 李四   │ F      │ 1945       │
│ clx3ghi...       │ 王五   │ M      │ 1968       │
└──────────────────┴────────┴────────┴────────────┘
```

### 4.2 什么是主键、外键

**主键（Primary Key）**：每行的唯一标识符，保证没有两行的主键相同。本项目所有表的主键都是 `id`，类型是 `cuid()`（一种随机生成的字符串 ID）。

**外键（Foreign Key）**：一张表里存了另一张表的主键，用来表达"关联关系"。

```
genealogies 表（族谱）：
  id = "gen001"
  owner_id = "usr001"   ← 这是外键，指向 users 表的 id
  name = "张氏家谱"

users 表（用户）：
  id = "usr001"         ← 这是主键
  username = "zhangsan"
```

外键约束的作用：如果 `users.id = "usr001"` 的行被删除，PostgreSQL 会自动处理（根据设置：拒绝删除，或者级联删除所有关联族谱）。本项目用的是 `onDelete: Cascade`（级联删除）。

### 4.3 什么是 Prisma Schema

`prisma/schema.prisma` 是一个特殊的配置文件（`.prisma` 格式，不是 TypeScript）。它描述：
- 用哪种数据库（PostgreSQL）
- 每张表有什么字段，什么类型
- 表之间的关系

Prisma 读取这个文件后可以做两件事：
1. **迁移**：在真实数据库中创建或修改表结构
2. **代码生成**：生成 TypeScript 类型，让你在代码中有类型提示

### 4.4 什么是「迁移」

迁移（Migration）是数据库结构变更的历史记录。每次修改 `schema.prisma` 后，执行：

```bash
npm run db:migrate
```

Prisma 会：
1. 对比当前 schema 和数据库实际结构
2. 生成一个 SQL 文件（如 `prisma/migrations/20240101_init/migration.sql`）
3. 执行这个 SQL 文件，修改数据库

这样做的好处是：迁移文件可以提交到 git，其他人拿到代码后执行一次迁移就能还原完整的数据库结构。

### 4.5 什么是 ER 图（实体关系图）

ER 图（Entity-Relationship Diagram，实体关系图）是描述数据库表和关系的图。本项目 6 张表的关系：

```
User（用户）
  │
  │ 一个用户可以拥有多个族谱（一对多）
  ▼
Genealogy（族谱）
  │
  │ 一个族谱可以有多个协作者（多对多，通过关联表）
  ├──→ GenealogyCollaborator ←── User
  │
  │ 一个族谱包含多个成员（一对多）
  ▼
Member（成员）
  │
  ├──→ ParentChild（亲子关系）
  │       ← Member（parent_id）→ Member（child_id）
  │       （成员自引用多对多）
  │
  └──→ Marriage（婚姻关系）
          ← Member（member1_id）→ Member（member2_id）
          （成员自引用多对多）
```

### 4.6 什么是数据库范式（3NF）

范式是数据库设计的规范，目的是**减少数据冗余**（同样的信息不存两份）。

第三范式（3NF）的要求：
1. 每列只存一个值（不把"张三,李四"存到同一列）
2. 每列依赖整个主键（不只依赖主键的一部分）
3. 非主键列不能依赖另一个非主键列

本项目满足 3NF 的例子：
- 成员的姓名存在 `members.name`，不在 `parent_child` 里重复存
- 族谱创建者的 username 不存在 `genealogies` 表里，而是通过 `owner_id` 外键查询得到

> **反例对比**：如果 `genealogies` 表里加了 `owner_username` 字段，那么用户改名时就要同时改两个地方（`users.username` 和 `genealogies.owner_username`），很容易不一致。通过外键引用，就只需要改一处。

### 4.7 什么是 CUID

CUID（Collision-resistant Unique Identifier，抗碰撞唯一标识符）是一种随机生成的字符串 ID，格式类似 `clx1a2b3c4d5e6f7g8h9`。

本项目所有表的主键都用 `@default(cuid())`，意思是插入新行时自动生成一个 CUID，不需要手动赋值。

与整数自增 ID（`@default(autoincrement())`）相比，CUID 的优点：
- 全球唯一，多个数据库合并不会冲突
- 不暴露总数量（看不出来系统里有多少用户）
- URL 路径中使用更安全（`/genealogies/clx123...` 比 `/genealogies/3` 更难猜测）

## 5. 逐步代码讲解

### 5.1 步骤 1：`.env` 环境变量 【共同】

#### A. 动机

数据库密码、密钥等敏感信息不能硬编码在代码里（否则提交到 git 后所有人都能看到）。惯例是把这些信息存在 `.env` 文件里，代码通过 `process.env.变量名` 读取。

`.env` 文件不提交到 git（`.gitignore` 里有），每台机器自己配置。

#### B. 你将要理解的文件

*文件 `.env`：*

```bash
# 数据库连接字符串
# 格式：postgresql://用户名:密码@主机:端口/数据库名
DATABASE_URL="postgresql://postgres:040604@localhost:5432/genealogyms"

# NextAuth 密钥（任意随机字符串，用于加密 Session）
AUTH_SECRET="your-secret-key-here"

# 本地开发地址
NEXTAUTH_URL="http://localhost:3000"
```

#### C. 这段配置做了什么

- `DATABASE_URL` 是 Prisma 连接 PostgreSQL 的字符串
  - `postgres` 是 PostgreSQL 的默认超级用户名
  - `040604` 是你安装时设置的密码
  - `localhost:5432` 是本机的默认 PostgreSQL 地址和端口
  - `genealogyms` 是数据库名称（需要提前创建）
- `AUTH_SECRET` 被 NextAuth 用来签名 JWT token（类似签名的密钥）
- `NEXTAUTH_URL` 告诉 NextAuth 当前应用运行在哪个地址

> **反例对比**：如果把密码直接写在 `lib/db.ts` 里（`connectionString = "postgresql://postgres:040604@..."`），一旦提交到 GitHub，任何人都能看到密码。`.env` 文件必须加入 `.gitignore`。

---

### 5.2 步骤 2：`prisma.config.ts` — Prisma 配置 【开发者甲】

#### A. 动机

Prisma 7 引入了新的配置文件 `prisma.config.ts`，把数据库 URL 从 `schema.prisma` 中分离出来。这样做是因为：数据库 URL 包含密码，不应该放在 schema 文件里（schema 文件通常会提交到 git）。

#### B. 完整代码

*文件 `prisma.config.ts`：*

```typescript
import "dotenv/config";
import { defineConfig } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
  },
  datasource: {
    url: process.env["DATABASE_URL"],
  },
});
```

#### C. 这段代码做了什么

- `import "dotenv/config"`：读取 `.env` 文件，把里面的变量注入到 `process.env` 中。这样 `process.env["DATABASE_URL"]` 就能拿到数据库 URL。
- `defineConfig({...})`：Prisma 7 的配置函数
  - `schema`：告诉 Prisma 在哪里找 schema 文件（`prisma/schema.prisma`）
  - `migrations.path`：迁移历史文件存放位置
  - `datasource.url`：数据库连接 URL，从环境变量读取

这个文件只在执行 `npm run db:migrate`、`npm run db:generate` 等 Prisma CLI 命令时使用，应用运行时不读这个文件（应用连接数据库由 `lib/db.ts` 负责）。

### 5.3 步骤 3：`prisma/schema.prisma` — 数据库模型定义 【开发者甲】

#### A. 动机

`schema.prisma` 是整个数据库设计的"蓝图"。它定义 6 张表，用 Prisma 特有的 Schema 语言写。写完后运行迁移命令，PostgreSQL 里就会出现对应的真实表。

#### B. 完整代码

*文件 `prisma/schema.prisma`：*

```prisma
// Prisma Schema — 寻根溯源族谱管理系统
// 数据库: PostgreSQL 16
// 范式: 所有表满足 3NF / BCNF

generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["driverAdapters"]
}

datasource db {
  provider = "postgresql"
}

// 用户表
model User {
  id        String   @id @default(cuid())
  username  String   @unique
  email     String   @unique
  password  String   // bcrypt hash
  createdAt DateTime @default(now()) @map("created_at")

  // 关系
  ownedGenealogies    Genealogy[]              @relation("GenealogyOwner")
  collaborations      GenealogyCollaborator[]

  @@map("users")
}

// 族谱表
model Genealogy {
  id          String   @id @default(cuid())
  name        String
  surname     String
  createdYear Int      @map("created_year")
  createdAt   DateTime @default(now()) @map("created_at")

  ownerId  String @map("owner_id")
  owner    User   @relation("GenealogyOwner", fields: [ownerId], references: [id], onDelete: Cascade)

  collaborators GenealogyCollaborator[]
  members       Member[]

  @@map("genealogies")
}

// 族谱协作者（多对多关联表：Genealogy ↔ User）
model GenealogyCollaborator {
  genealogyId String   @map("genealogy_id")
  userId      String   @map("user_id")
  invitedAt   DateTime @default(now()) @map("invited_at")

  genealogy Genealogy @relation(fields: [genealogyId], references: [id], onDelete: Cascade)
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@id([genealogyId, userId])
  @@map("genealogy_collaborators")
}

// 家族成员表
model Member {
  id          String  @id @default(cuid())
  genealogyId String  @map("genealogy_id")
  name        String
  gender      String  // 'M' | 'F'
  birthYear   Int?    @map("birth_year")
  deathYear   Int?    @map("death_year")
  bio         String?
  generation  Int?

  genealogy Genealogy @relation(fields: [genealogyId], references: [id], onDelete: Cascade)

  parentRelations ParentChild[] @relation("ChildRelation")
  childRelations  ParentChild[] @relation("ParentRelation")
  marriagesAsP1   Marriage[]    @relation("MarriageMember1")
  marriagesAsP2   Marriage[]    @relation("MarriageMember2")

  @@index([genealogyId])
  @@index([name])
  @@map("members")
}

// 亲子关系表（成员自引用多对多）
model ParentChild {
  parentId String @map("parent_id")
  childId  String @map("child_id")

  parent Member @relation("ParentRelation", fields: [parentId], references: [id], onDelete: Cascade)
  child  Member @relation("ChildRelation", fields: [childId], references: [id], onDelete: Cascade)

  @@id([parentId, childId])
  @@index([parentId])
  @@index([childId])
  @@map("parent_child")
}

// 婚姻关系表（成员自引用多对多）
model Marriage {
  id           String @id @default(cuid())
  member1Id    String @map("member1_id")
  member2Id    String @map("member2_id")
  marriageYear Int?   @map("marriage_year")
  divorceYear  Int?   @map("divorce_year")

  member1 Member @relation("MarriageMember1", fields: [member1Id], references: [id], onDelete: Cascade)
  member2 Member @relation("MarriageMember2", fields: [member2Id], references: [id], onDelete: Cascade)

  @@index([member1Id])
  @@index([member2Id])
  @@map("marriages")
}
```

#### C. 这段代码做了什么（逐块解析）

**块 1：`generator client`**

```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["driverAdapters"]
}
```

告诉 Prisma 生成 JavaScript/TypeScript 版本的客户端代码（放在 `node_modules/@prisma/client`）。

`previewFeatures = ["driverAdapters"]` 是 Prisma 7 的新特性，启用「驱动适配器」功能——允许我们用 `@prisma/adapter-pg`（pg 库的适配器）代替 Prisma 自带的数据库驱动，这在 Next.js 服务器环境中更稳定。

**块 2：`datasource db`**

```prisma
datasource db {
  provider = "postgresql"
}
```

告诉 Prisma 使用 PostgreSQL。注意这里**没有** `url = env("DATABASE_URL")`——因为 Prisma 7 把 URL 配置移到了 `prisma.config.ts`，不再写在 schema 文件里。

**块 3：`model User`（用户表）**

```prisma
model User {
  id        String   @id @default(cuid())
  username  String   @unique
  email     String   @unique
  password  String
  createdAt DateTime @default(now()) @map("created_at")
  ...
  @@map("users")
}
```

- `@id @default(cuid())`：主键，自动生成 CUID
- `@unique`：唯一约束，插入重复用户名/邮箱会报错
- `@default(now())`：插入时自动填入当前时间
- `@map("created_at")`：Prisma 代码里用 `createdAt`（驼峰），数据库表里实际是 `created_at`（下划线），`@map` 做映射
- `@@map("users")`：Prisma 模型名是 `User`，数据库表名是 `users`（PostgreSQL 惯例用小写）

**块 4：`model Member`（成员表）中的自引用关系**

```prisma
parentRelations ParentChild[] @relation("ChildRelation")
childRelations  ParentChild[] @relation("ParentRelation")
```

这是最难理解的部分。一个成员既可以是「子女」（有父母），又可以是「父母」（有子女）。`ParentChild` 表通过两个字段 `parentId` 和 `childId` 连接同一张 `members` 表：

```
members 表（成员）：  id = "A"（张三）
                      id = "B"（张三之子）

parent_child 表：
  parent_id = "A"（张三）
  child_id  = "B"（张三之子）
```

`@relation("ChildRelation")` 和 `@relation("ParentRelation")` 是命名标签，告诉 Prisma「这个字段对应 ParentChild 表中的哪一侧」，避免歧义。

**块 5：`model GenealogyCollaborator`（协作者关联表）**

```prisma
@@id([genealogyId, userId])
```

`@@id([...])` 表示**复合主键**：用 `genealogyId` 和 `userId` 的组合作为主键。好处：同一个族谱不能邀请同一个用户两次（因为组合主键不允许重复）。

**块 6：`@@index` 索引**

```prisma
@@index([genealogyId])
@@index([name])
```

索引是一种数据结构，让数据库能快速查找特定列的值。没有索引时，数据库要扫描整张表（全表扫描）。有了索引，查找速度从 O(n) 降为 O(log n)。

`@@index([genealogyId])` 让「按族谱ID查询成员」快很多（这是最常用的查询之一）。

### 5.4 步骤 4：`lib/db.ts` — Prisma 客户端单例 【开发者甲】

#### A. 动机

数据库连接是昂贵的资源，不能每次请求都新建一个连接。`lib/db.ts` 实现了「单例模式」：整个应用只有一个 `PrismaClient` 实例，所有代码共用这一个实例。

#### B. 完整代码

*文件 `lib/db.ts`：*

```typescript
import { PrismaClient } from "@prisma/client";
import { PrismaPg } from "@prisma/adapter-pg";

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };

function createPrismaClient() {
  const connectionString = process.env.DATABASE_URL!;
  const adapter = new PrismaPg({ connectionString });
  return new PrismaClient({
    adapter,
    log: process.env.NODE_ENV === "development" ? ["error", "warn"] : ["error"],
  });
}

export const prisma = globalForPrisma.prisma ?? createPrismaClient();

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
```

#### C. 这段代码做了什么

**为什么需要单例？**

在开发模式下，Next.js 会「热重载」（修改代码后自动重新编译）。如果每次重载都创建一个新的 `PrismaClient`，很快就会出现「连接数超限」错误（PostgreSQL 默认最多 100 个并发连接）。

**单例的实现原理：**

```typescript
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };
```

`globalThis` 是全局对象（类似浏览器里的 `window`），热重载不会重置它。把 `prisma` 存在这里，下次模块重载时还能找到同一个实例。

```typescript
export const prisma = globalForPrisma.prisma ?? createPrismaClient();
```

`??` 是「空值合并运算符」：如果 `globalForPrisma.prisma` 已经存在，直接用它；否则才创建新的。

**`PrismaPg` 适配器是什么？**

```typescript
const adapter = new PrismaPg({ connectionString });
return new PrismaClient({ adapter, ... });
```

Prisma 7 需要显式提供一个「数据库驱动适配器」。`@prisma/adapter-pg` 是基于 `pg` 库（Node.js 的 PostgreSQL 驱动）封装的适配器。在 Next.js 环境中使用它比 Prisma 自带的驱动更稳定。

**`log` 配置：**

- 开发环境：打印 `error` 和 `warn` 日志（方便调试）
- 生产环境：只打印 `error`

> **反例对比**：如果直接 `export const prisma = new PrismaClient()`，每次模块重载都创建新实例，开发过程中几分钟就会出现「too many connections」错误。把实例挂在 `globalThis` 上是 Prisma 官方推荐的 Next.js 单例模式。

## 6. 端到端走查：新建一条成员记录

用「开发者甲调用 `prisma.member.create(...)` 插入一个新成员」来追踪完整链路：

| 步骤 | 位置 | 发生了什么 |
|------|------|-----------|
| 1 | `lib/db.ts` | 应用启动时，`createPrismaClient()` 建立连接池，`prisma` 实例就绪 |
| 2 | `app/api/members/route.ts` | 前端 POST 请求到达，route.ts 里调用 `prisma.member.create({...})` |
| 3 | `@prisma/client` | Prisma 把 TypeScript 对象翻译成 SQL：`INSERT INTO members (id, name, ...) VALUES (...)` |
| 4 | `@prisma/adapter-pg` | 适配器把 SQL 语句传给 `pg` 库 |
| 5 | `pg` 库 | 通过 TCP 连接把 SQL 发送到 PostgreSQL 进程 |
| 6 | PostgreSQL | 执行 INSERT，返回插入的行 |
| 7 | 反向路径 | 数据从 PG → pg → adapter → Prisma → TypeScript 对象返回给 route.ts |
| 8 | `route.ts` | 把返回的成员对象序列化成 JSON，通过 HTTP 返回给浏览器 |

## 7. 数据库表关系完整总结

```
┌─────────────────────────────────────────────────────────────────┐
│                         6 张表的关系                              │
│                                                                   │
│  users (用户)                                                     │
│    id ─────────────────┐                                         │
│    username             │  外键 owner_id                         │
│    email                ▼                                        │
│    password        genealogies (族谱)                            │
│    created_at        id ─────────────┐                          │
│         │            name            │  外键 genealogy_id        │
│         │            surname         ▼                           │
│         │            owner_id   members (成员)                   │
│         │            created_at   id ─────┐─────┐               │
│         │                │         name   │     │               │
│         │                │         gender │     │               │
│         │   外键          │         birth_year   │               │
│         └──────────────► │         death_year   │               │
│   genealogy_collaborators│         generation   │               │
│     genealogy_id ────────┘              │       │               │
│     user_id ──────────────►users        │       │               │
│     invited_at                          │       │               │
│                               parent_child  marriages           │
│                                parent_id    member1_id          │
│                                child_id     member2_id          │
│                                (两个外键      marriage_year      │
│                                 都指向        divorce_year       │
│                                 members.id)                     │
└─────────────────────────────────────────────────────────────────┘
```

用一句话概括每张表的职责：

| 表名 | 存什么 | 和谁有关系 |
|------|--------|-----------|
| `users` | 账号信息（用户名、密码哈希） | 拥有族谱 |
| `genealogies` | 族谱信息（名称、姓氏） | 属于某个用户 |
| `genealogy_collaborators` | 协作权限（谁能编辑哪个族谱） | 用户 ↔ 族谱 |
| `members` | 成员个人信息 | 属于某个族谱 |
| `parent_child` | 亲子血缘关系（谁是谁的父/母） | 成员 ↔ 成员 |
| `marriages` | 婚姻关系（谁和谁结婚） | 成员 ↔ 成员 |

## 8. 本章小结

| 概念 | 一句话记忆 |
|------|-----------|
| `schema.prisma` | 数据库蓝图，定义所有表和关系 |
| `prisma.config.ts` | 告诉 Prisma CLI 连哪个数据库 |
| `lib/db.ts` | 单例模式，整个应用共用一个数据库连接 |
| `@id @default(cuid())` | 主键，自动生成随机字符串ID |
| `@map("snake_case")` | TypeScript 用驼峰命名，数据库用下划线命名 |
| `onDelete: Cascade` | 删除父记录时自动删除所有关联子记录 |
| `@@id([a, b])` | 复合主键，两个字段的组合唯一 |
| `@@index([col])` | 为列建索引，加速查询 |
| 迁移（Migration） | schema 变更的历史记录，用 `npm run db:migrate` 执行 |
