# 项目理解指南 1 — 项目总览与技术栈

## 0. 前言

这是「寻根溯源族谱管理系统」项目理解系列的第一章。本章不涉及任何具体代码，只讲清楚两件事：**整个系统是什么**、**它由哪些技术拼在一起**。读完本章，你应当能够：

1. 用一句话向老师解释这个系统是干什么的
2. 描述系统的技术栈，并说出每项技术负责哪一层
3. 看到任何一个源码文件时，知道它大概属于哪个层次、由哪位开发者负责

---

## 1. 这个系统是什么

「寻根溯源」是一个**多用户在线族谱管理系统**，运行在浏览器中，数据存储在 PostgreSQL 数据库中。

系统的核心功能：

- 用户注册/登录，每人管理自己的族谱，也可以邀请他人协作
- 为家族添加成员，记录姓名、性别、生卒年、生平
- 记录成员间的亲子关系和婚姻关系
- 以树状图展示家族层级
- 输入一个成员 ID，用**递归 SQL** 查出所有历代祖先
- 输入两个成员 ID，用**图搜索算法**找出两人之间的亲缘路径

## 2. 技术栈总览

### 2.1 技术栈是什么意思

「技术栈」就是完成一个项目所用到的所有技术的组合。就像盖房子需要地基、砖墙、屋顶三层材料，软件项目也有分层的技术：

```
┌─────────────────────────────────────────┐
│          浏览器 (用户看到的界面)           │  ← 前端 / Frontend
│   React 组件 + Tailwind CSS 样式          │
├─────────────────────────────────────────┤
│          Next.js 服务器 (中间层)           │  ← 全栈框架 / Full-Stack
│   API 路由 + 页面渲染 + 用户认证            │
├─────────────────────────────────────────┤
│           Prisma ORM (翻译层)             │  ← 数据访问层
│   把 TypeScript 对象翻译成 SQL 语句         │
├─────────────────────────────────────────┤
│        PostgreSQL 数据库 (底层存储)         │  ← 数据库 / Database
│   表格形式持久化存储所有族谱数据              │
└─────────────────────────────────────────┘
```

### 2.2 每项技术的职责

| 技术 | 版本 | 职责 | 由谁负责 |
|------|------|------|---------|
| **Next.js** | 16 | 全栈框架：把前端页面和后端 API 放在同一个项目里 | 共同 |
| **TypeScript** | 5 | JavaScript 的超集，加了类型检查，写错参数会报错 | 共同 |
| **PostgreSQL** | 18 | 关系型数据库，存放所有用户、族谱、成员数据 | 开发者甲 |
| **Prisma** | 7 | ORM（对象关系映射），用 TypeScript 代码操作数据库而不用手写 SQL | 开发者甲 |
| **NextAuth.js** | v5 | 处理用户登录/注销的认证库，管理会话（Session） | 开发者甲 |
| **Tailwind CSS** | 4 | 工具类 CSS 框架，在 HTML 标签上直接写样式类名 | 开发者乙 |
| **ECharts** | 6 | Apache 的图表库，用来渲染树形图和关系图 | 开发者乙 |
| **bcryptjs** | 3 | 密码哈希库，把明文密码加密后存入数据库 | 开发者甲 |

## 4. 核心概念快速理解

### 4.1 Next.js 的「App Router」是什么

Next.js 是一个全栈框架，意思是**前端界面**和**后端接口**写在同一个项目里，不需要分成两个独立项目。

**App Router** 是 Next.js 13+ 引入的路由方式。规则非常简单：**文件夹的名字就是 URL 路径**。

```
app/
├── dashboard/page.tsx    → 访问 /dashboard 时渲染这个文件
├── genealogies/page.tsx  → 访问 /genealogies 时渲染这个文件
└── genealogies/[id]/     → [id] 是动态段，访问 /genealogies/abc123 时
    └── page.tsx            id = "abc123"
```

`api/` 下的文件夹也遵循同样规则，但里面的文件叫 `route.ts` 而不是 `page.tsx`：

```
app/api/members/route.ts  → 访问 /api/members 时执行这里的代码（不渲染页面，返回 JSON）
```

### 4.2 「路由组」是什么

文件夹名用括号包裹时（如 `(auth)`、`(main)`），它**不影响 URL 路径**，只是用来把页面分组、共享同一个布局：

- `(auth)/login/page.tsx` → URL 仍是 `/login`（不是 `/auth/login`）
- `(main)/dashboard/page.tsx` → URL 仍是 `/dashboard`

分组的好处是：`(main)` 文件夹里有 `layout.tsx`，所有 `(main)` 下的页面都会自动套上侧边栏布局，而 `(auth)` 下的登录页不会有侧边栏。

### 4.3 什么是 API Route（后端接口）

`app/api/` 下的每个 `route.ts` 文件都是一个**后端接口**（类似 SpringBoot 中的 `@RestController`）。

文件内导出的函数名对应 HTTP 方法：

```typescript
// app/api/genealogies/route.ts
export async function GET() { ... }    // 处理 GET /api/genealogies
export async function POST() { ... }   // 处理 POST /api/genealogies
```

前端页面通过 `fetch("/api/genealogies")` 调用这些接口，拿到 JSON 数据。

### 4.4 Prisma 是什么，为什么不直接写 SQL

Prisma 是一个 **ORM（对象关系映射）**工具。它让你用 TypeScript 代码操作数据库，而不是手写 SQL 字符串。

对比示例：

```typescript
// 不用 Prisma（手写 SQL，容易出错）：
const rows = await db.query("SELECT * FROM genealogies WHERE owner_id = $1", [userId]);

// 用 Prisma（TypeScript 对象风格，有类型提示）：
const genealogies = await prisma.genealogy.findMany({
  where: { ownerId: userId }
});
```

Prisma 的好处：
- 写错字段名时，TypeScript 立刻报错（而不是运行时才发现）
- 自动防止 SQL 注入攻击
- 自动管理数据库连接

但对于**复杂的递归查询**（树形祖先查询），Prisma 不够表达，这时候项目直接用 `prisma.$queryRaw` 执行原生 SQL，兼顾两种方式的优点。

### 4.5 「服务端组件」和「客户端组件」的区别

Next.js 有两种组件：

| 类型 | 文件开头 | 在哪里运行 | 可以做什么 |
|------|---------|-----------|-----------|
| **服务端组件** | 默认（无需标注） | 服务器上 | 直接查数据库、不能用 `useState`/`useEffect` |
| **客户端组件** | `"use client"` | 浏览器中 | 有交互（按钮点击、表单输入）、不能直接查数据库 |

本项目的规律：
- `app/(main)/dashboard/page.tsx` 是**服务端组件**，直接 `await prisma.xxx` 查数据库
- `app/(main)/genealogies/page.tsx` 是**客户端组件**（有 `"use client"`），通过 `fetch("/api/...")` 调接口

## 5. 一次完整的用户请求流程

以「用户打开成员管理页面」为例，追踪从浏览器到数据库的完整链路：

```
浏览器访问 /genealogies/abc123/members
            │
            ▼
Next.js 路由匹配
  └─ 找到 app/(main)/genealogies/[id]/members/page.tsx
            │
            ▼
proxy.ts（路由守卫）检查登录状态
  ├─ 未登录 → 重定向到 /login
  └─ 已登录 → 继续
            │
            ▼
app/(main)/layout.tsx 渲染侧边栏框架
            │
            ▼
members/page.tsx 在浏览器中执行：
  └─ useEffect → fetch("/api/members?genealogyId=abc123")
            │
            ▼
app/api/members/route.ts 的 GET 函数执行：
  1. 检查 session（auth()）确认登录状态
  2. 检查权限（用户是否有权访问此族谱）
  3. 调用 prisma.member.findMany({ where: { genealogyId: "abc123" } })
            │
            ▼
Prisma 把上面的代码翻译成 SQL：
  SELECT * FROM members WHERE genealogy_id = 'abc123'
            │
            ▼
PostgreSQL 执行 SQL，返回数据行
            │
            ▼
Prisma 把数据行转成 TypeScript 对象数组
            │
            ▼
route.ts 把对象数组序列化成 JSON，通过 HTTP 返回给浏览器
            │
            ▼
members/page.tsx 收到 JSON，调用 setMembers() 更新状态
            │
            ▼
React 重新渲染，表格显示成员列表
```

## 6. 两位开发者的分工说明

演示时，两位开发者应能分别清楚说明自己的工作：

### 开发者甲（数据库与后端）的工作

1. **数据库建模**：设计 6 张表的结构（`schema.prisma`），确定主键、外键、约束
2. **数据库迁移**：执行 `npm run db:migrate` 在 PostgreSQL 中创建真实的表
3. **所有 API 接口**：`app/api/` 下的所有 `route.ts` 文件，处理增删改查请求
4. **认证系统**：配置 NextAuth（`lib/auth.ts`），实现 JWT 登录验证
5. **复杂 SQL 查询**：写递归 CTE（祖先查询）、BFS 亲缘路径（`lib/queries/`）
6. **数据生成**：编写脚本生成 10 万+测试数据（`scripts/seed.ts`）

### 开发者乙（前端与可视化）的工作

1. **UI 组件库**：手工实现 Button、Input、Card、Modal 等基础组件（`components/ui/`）
2. **页面实现**：所有用户可见的页面（`app/(main)/` 和 `app/(auth)/`）
3. **侧边栏导航**：实现带路由高亮的侧边栏（`components/ui/sidebar.tsx`）
4. **树形可视化**：集成 ECharts，把递归查询结果渲染成树状图（`components/tree/`）
5. **成员管理界面**：成员表格、模糊搜索、分页、ID 复制弹窗（`members/page.tsx`）
6. **亲缘通路可视化**：把 BFS 路径结果渲染成圆形节点路径图（`relation/page.tsx`）

## 7. 本章小结

| 概念 | 一句话记忆 |
|------|-----------|
| Next.js App Router | 文件夹名就是 URL，`page.tsx` 是页面，`route.ts` 是 API |
| 路由组 `(main)` `(auth)` | 括号文件夹不影响 URL，只用来共享布局 |
| Prisma ORM | 用 TypeScript 对象操作数据库，自动翻译成 SQL |
| 服务端组件 | 默认，在服务器运行，可直接查数据库 |
| 客户端组件 | `"use client"` 开头，在浏览器运行，有交互逻辑 |
| API Route | `app/api/` 下的 `route.ts`，类似 SpringBoot 的 Controller |
| PostgreSQL | 存所有数据的关系型数据库，本项目用 18 版本 |
