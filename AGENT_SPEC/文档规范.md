# 项目理解指南文档规范（GUIDE-SPEC）

本文档记录「寻根溯源族谱管理系统」项目理解指南系列的写作规范，供后续 AI 或人工编写同系列文档时参考对齐。

---

## 一、总体定位

指南系列面向**完全没有相关技术栈开发经验的初学者**（具体背景：少量 React/Java SpringBoot 经验，熟悉 OOP 理论，读过数据库教材但无实操经验）。

文档的核心目标不是"告诉读者怎么做"，而是"帮助读者理解为什么这样做、是什么在运行、会发生什么"，使其能在演示和问答中自信作答。

---

## 二、文档分工与顺序

全系列共 6 份指南，按「由底层到高层、由后端到前端」的教程性顺序排列：

| 编号 | 文件名 | 核心内容 | 主要归属开发者 |
|------|--------|---------|--------------|
| 1 | 项目理解指南1-项目总览与技术栈.md | 整体架构、目录树、请求流程 | 共同理解 |
| 2 | 项目理解指南2-数据库设计.md | 数据建模、Prisma、迁移、ER图 | 开发者甲 |
| 3 | 项目理解指南3-用户认证系统.md | NextAuth、注册/登录API、路由守卫 | 甲（API）+ 乙（页面）|
| 4 | 项目理解指南4-族谱与成员管理.md | CRUD API + 对应前端页面 | 甲（API）+ 乙（页面）|
| 5 | 项目理解指南5-树形可视化与查询.md | ECharts、递归CTE、BFS亲缘路径 | 甲（SQL）+ 乙（图表）|
| 6 | 项目理解指南6-数据工程与SQL汇总.md | 数据生成脚本、COPY、索引、统计SQL | 开发者甲 |

**两位开发者的职责划分（用于演示答辩）：**

- **开发者甲（数据库与后端）**：负责数据库建模（ER图、schema.prisma）、所有 API Route 文件、SQL 查询函数（lib/queries/）、NextAuth 配置、数据生成脚本（scripts/）、索引优化
- **开发者乙（前端与可视化）**：负责所有页面文件（app/(main)/、app/(auth)/）、UI 组件（components/）、ECharts 树形图组件、侧边栏、成员表单

## 三、每份文档的结构模板

```
## 0. 前言
本章目标、与前章的衔接、本章结束后读者能理解什么

## 1. 本章目标
1.1 功能目标（这部分代码实现了什么用户可感知的功能）
1.2 工程目标（技术上做了什么，引入了哪些文件/概念）

## 2. 先运行起来（Smoke Run）
给出最短的操作步骤，让读者在阅读细节前先看到效果

## 3. 文件结构与变更总览
3.1 涉及的文件目录（ASCII 树，标注新增/修改）
3.2 数据/请求流程图（ASCII 箭头图）
3.3 编码步骤索引表（#, 文件, 新增/修改, 一句话说明）

## 4. 基础知识铺垫
用通俗语言解释本章涉及的**所有新概念**，确保读者无需查外部资料

## 5. 逐步代码讲解
每个步骤格式：
### 5.N 步骤 N：文件名
#### A. 动机（为什么要写这段代码）
#### B. 你将要理解的操作
#### C. 完整代码（原样抄录源码）
#### D. 这段代码做了什么（逐行/逐块解释）
#### E. 验证方式（如何确认这段代码工作正常）

## 6. 端到端串联走查
用一条具体用户操作（如"点击登录"）追踪完整的调用链，
以表格或箭头图的形式，列出每个跳转点、文件位置、关键代码

## 7. 常见问题与反例对比
列出初学者容易误解的点，每条用"反例 → 正确做法 → 原因"结构

## 8. 本章小结与知识点回顾
以要点列表形式总结本章新知识
```

## 四、写作风格细则

### 4.1 代码块规范

- 每个代码块开头注明文件路径（斜体或注释形式）：
  ```
  *文件 `lib/db.ts`：*
  ```
- 长文件完整抄录，不做省略（读者需要对照阅读源码）
- 关键行在代码块下方用「D. 这段代码做了什么」解释，不在代码块内加干扰注释

### 4.2 ASCII 图规范

- 目录树用 `├─`、`└─`、`│` 构成，标注 `# 新增` / `# 修改`
- 流程图用 `→`、`↓`、`│`、`▼`、`└─` 构成
- 表示层次的缩进用空格，不用 Tab

### 4.3 术语解释规范

- 每个专业术语（如「递归 CTE」「JWT」「ORM」）首次出现时用括号给出中文解释
- 重要概念单独开一个 `### 4.N` 小节，包含：是什么、为什么要用它、最小示例代码

### 4.4 「反例对比」规范

每个重要设计决策（如「为什么用 Prisma 而不是直接写 SQL」）提供：

```
> **反例对比**：如果 [这样做]，会出现 [这个问题]，因为 [原因]。
> 本项目用 [正确做法]，是因为 [理由]。
```

### 4.5 开发者归属标注

在每个步骤标题旁标注归属，如：

```
### 5.3 步骤 3：lib/auth.ts — NextAuth 配置 【开发者甲】
```

---

## 五、语言规范

- 使用**中文**写作，技术术语保留英文原名（如 `PrismaClient`、`useEffect`），首次出现时给中文解释
- 句子不超过 40 字，复杂概念拆分成多句
- 避免「很简单」「很容易」「显然」等评价性词语
- 每个段落聚焦一个概念，不混合多个知识点

---

## 六、禁止事项

- 不省略代码（即使很长，也要完整抄录）
- 不写「参见 xxx 文档」（读者应该在本文档内找到所有答案）
- 不假设读者知道某个概念（哪怕是「什么是 JSON」也可以简要解释）
- 不使用 emoji
